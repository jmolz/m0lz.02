# Feature: PRDv2 Phase 7 — Background Execution + Live Status & Log Streaming (Revision 2)

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.
> **Tier 3 plan. Revision 2** applies all 11 Critical findings from Cycle 1 adversarial review (Claude architect + Codex GPT-5.4 xhigh). Subscription transport simplified per Codex reasoning divergence (initial snapshot IS the subscribe RPC response; no unsubscribe RPC). `run_id` + `Queued` status introduced to resolve same-feature collision + dispatch-window crash semantics. `max_global_provider_concurrency` split from `max_parallelism` to prevent intra-feature starvation. `--stream-json` replaces `--json` JSONL repurposing. DecisionSource scope cut to the genuine Phase 7 consumer only; circular dependency 11↔18 broken. Remaining Consider items applied silently in revision.

## Overview

Enable `pice evaluate --background` / `pice execute --background` to return immediately with `{feature_id, run_id}` while the daemon runs the full Stack Loops pipeline in a detached tokio task. Ship `pice status` (list / detail / `--follow` / `--wait`) and `pice logs {feature} [--layer L] [--follow]` for monitoring. Deliver desktop notifications on completion and pending-gate events via `notify-rust`. Support multiple concurrent evaluations bounded by a **dedicated global provider semaphore** (separate from intra-cohort `max_parallelism`). Preserve every v0.1–v0.6 invariant (manifest as source of truth, audit-before-manifest ordering, per-manifest lock serialization, fail-closed status).

**Architecture simplification (Cycle 1 divergence):** `manifest/subscribe` and `logs/stream` are single-request RPCs whose **response body IS the initial snapshot**. The connection then carries only `manifest/event` / `logs/chunk` notifications until the client closes it — no `unsubscribe` RPC, no subscription IDs, no concurrent read/write complexity. This drops `ManifestEvent::Snapshot` as a pseudo-event and eliminates `SubscriptionRegistry` keys in favor of per-connection cleanup.

## User Story

As a **developer running long multi-layer PICE evaluations**, I want to **dispatch evaluations in the background and monitor them from any terminal session (or CI job)**, so that **I keep working during a 4-minute 7-layer evaluation, receive a desktop notification when it completes or needs a gate decision, and can reliably pipe `pice evaluate --background --wait` into my CI scripts with contract-based exit codes**.

## Success Criteria

- [ ] `pice evaluate --background` returns with `{"feature_id": "...", "run_id": "...", "status": "background-dispatched"}` on stdout, exit 0. **Dispatch-return-time is a measured p95 SLO < 500ms over 50 dispatches** (not a hard per-call bound — pre-flight cost varies by filesystem); the hard invariant is "dispatch returns before the orchestrator blocks on any provider call."
- [ ] `pice execute --background` mirrors the same contract.
- [ ] The daemon runs the evaluation in a detached tokio task that outlives the originating RPC connection; dropping the CLI connection does NOT cancel the task.
- [ ] Re-dispatching an already-running `feature_id` returns `ExitJsonStatus::FeatureAlreadyRunning` (exit 1) with the existing `run_id` in the error payload. No replace / queue semantics in v0.7.
- [ ] `pice status` (no args) lists every non-pruned feature manifest with per-layer progress + overall status + current cost.
- [ ] `pice status {feature_id}` prints a detailed per-layer view including pending gates and the `pice review-gate` action hint.
- [ ] `pice status --follow` subscribes to `manifest/event` notifications for all features; renders live updates via TTY erase+rewrite when stdout is a terminal.
- [ ] `pice status --stream-json` (NEW flag, distinct from `--json`) emits NDJSON — one `ManifestEventPayload` per line. `--json` continues to emit a single JSON object (snapshot) and is **rejected with a clap validation error when paired with `--follow`**. This preserves the repo-wide invariant that `--json` emits exactly one JSON object to stdout.
- [ ] `pice status --wait {feature_id} [--timeout-secs N]` blocks until the feature reaches a terminal state (`Passed` / `Failed` / `PendingReview` with no remaining retries); exits 0 / 2 / 3 matching `pice evaluate` semantics. Times out → exits 4 (`ExitJsonStatus::WaitTimeout`).
- [ ] `pice logs {feature_id}` prints the captured provider session transcript for every layer, ordered by DAG topology; `--layer L` filters; `--follow` streams new chunks. A terminal `logs/chunk { terminal: true, reason }` frame explicitly signals end-of-stream to follow subscribers.
- [ ] Desktop notifications fire on `feature_complete` (success + failure variants) and `gate_requested` events via **`notify-rust` exclusively** on macOS/Linux/Windows; on `notify-rust` error, fall back to `eprintln!` + bell char (no per-platform shell-out re-implementation).
- [ ] Multiple concurrent background evaluations run without manifest-lock contention. Two separate limits govern concurrency:
  - `workflow.defaults.max_parallelism` (existing) bounds INTRA-feature cohort parallelism; capped at `MAX_PARALLELISM_HARD_CAP = 16`.
  - `workflow.defaults.max_global_provider_concurrency` (NEW) bounds TOTAL concurrent provider sessions across ALL features; default = `max_parallelism`, capped at `MAX_GLOBAL_PROVIDER_CONCURRENCY_HARD_CAP = 32`.
- [ ] `pice evaluate --background --wait` is observationally equivalent to synchronous mode for CI: exit 0 on pass, exit 2 on contract failure, exit 3 on pending gate, exit 4 on timeout. `--wait` is a field on `EvaluateRequest`/`ExecuteRequest`, plumbed end-to-end.
- [ ] Daemon startup reconciliation distinguishes two states:
  - `overall_status == Queued` (manifest written but task never acquired semaphore) → **delete the manifest** (it represents a dispatch that didn't produce work).
  - `overall_status == InProgress` (orchestrator was running) → rewrite as `Failed` with `halted_by = "failed-interrupted"`.
- [ ] `ManifestStatus::Queued` is a distinct variant. Dispatch writes `Queued` atomically before calling `FeatureJobManager::spawn`. The spawned task's FIRST action after acquiring the global semaphore permit is transitioning to `InProgress` + event emission.
- [ ] Job closures capture a `JobEnv` snapshot (`PICE_STATE_DIR`, project_root, parsed workflow snapshot, resolved contract paths) at spawn time. The daemon's live env changing later does NOT affect already-running jobs.
- [ ] `SubscribedGateSource` (the Phase 7 consumer) wires into `pice status --follow` interactive mode, prompting for gate decisions via stderr, firing `ReviewGate::Decide` RPC. Phase 6's foreground evaluate stdin-read path is **not touched** — the circular dependency is broken by cutting the foreground DecisionSource wiring entirely.
- [ ] Every new RPC method (`manifest/subscribe`, `logs/stream`) has **typed request + response DTOs** in `pice-core::protocol`, roundtrip tests, AND a daemon integration test exercising the wire protocol.
- [ ] No regression in Phase 4 adaptive halting, Phase 5 cohort parallelism + cancellation, or Phase 6 gate audit-before-manifest ordering.
- [ ] All validation commands pass with zero errors; test count ≥ 970 Rust + 96 TypeScript (baseline 836 + ~135 new after Cycle 2 integration tests).

```rust
    // writer and streams notifications until the client closes it.

    methods::MANIFEST_SUBSCRIBE => subscribe::manifest(ctx, conn, &request).await,
    methods::LOGS_STREAM        => subscribe::logs(ctx, conn, &request).await,
    _ => DaemonResponse::error(request.id, -32601, "method not found"),
}
```

**Subscribe transport shape:**

```jsonc
// CLI → daemon
{"jsonrpc":"2.0","id":7,"method":"manifest/subscribe","auth":"…","params":{"feature_id":"auth-20260421"}}

// daemon → CLI (single response carrying the initial snapshot)
```

**Event emission convention — enforced by the `ManifestSaver` trait:**

```rust
// Only `ManifestSaver::save(&self, manifest, path)` is callable at orchestrator
// sites. Its only impl holds `&EventBus`; a save WITHOUT an emit is a
// compile error, not a grep-miss.
trait ManifestSaver {
    fn save(&self, manifest: &VerificationManifest, path: &Path) -> Result<()>;
}

struct EventEmittingSaver<'a> { bus: &'a EventBus }
impl<'a> ManifestSaver for EventEmittingSaver<'a> {
    fn save(&self, manifest: &VerificationManifest, path: &Path) -> Result<()> {
        manifest.save(path)?;                                  // crash-safe fsync+rename+fsync-dir
        self.bus.emit_manifest_state(manifest);                // post-save, infallible-log
        Ok(())
    }
}
```

The `EventBus::emit_*` helpers are infallible — `broadcast::Sender::send` returns `Err` only when there are zero receivers, the normal case when no adapter is subscribed. Never propagate; `tracing::trace!` at most.

Key behaviors:
1. **Dual-semaphore architecture** — the manager holds ONE `Arc<Semaphore>` representing `max_global_provider_concurrency` (clamped to `MAX_GLOBAL_PROVIDER_CONCURRENCY_HARD_CAP = 32`). 2. **Canonical semaphore lifecycle** (Cycle 2 clarification — single authoritative flow):
   - Dispatch handler writes `ManifestStatus::Queued` to disk.
   - Dispatch handler calls `FeatureJobManager::spawn(...)`, which **immediately** `tokio::spawn`s the future without acquiring any permit (spawn returns within microseconds).
   - The spawned future's FIRST action is `global_sem.acquire_owned().await`.
   - Once the permit is held, the future transitions the manifest `Queued → InProgress` via `saver.save_and_emit(..., SaveIntent::LayerStarted { layer })`.

**Background + wait (`background: true, wait: true`) — two-phase CLI flow (Cycle 2 clarification):**
1. Phase A — dispatch: daemon handler writes `Queued`, spawns the job, returns `CommandResponse::Json { value: {feature_id, run_id, status: "background-dispatched"} }` within the SLO. The dispatch RPC connection closes immediately after return.
2. Phase B — wait: the CLI (not the daemon handler) then opens a SECOND connection and sends a `manifest/subscribe` RPC filtered to `{feature_id}`. It consumes events until a terminal event arrives (`FeatureComplete` or `Cancelled`) OR `timeout_secs` elapses. If the initial snapshot response ALREADY shows a terminal state (race: daemon finished between dispatch and subscribe), the CLI exits immediately with the correct code — it does NOT hang waiting for a future event (Codex Cycle 2 terminal-state short-circuit fix).
3. 4. **`--wait` is NOT restart-safe** (honest correction of the prior claim): if the daemon dies mid-wait, the CLI receives a socket-close error. The background task is terminated; startup reconciliation rewrites the manifest to `Failed(failed-interrupted)`. The CLI surfaces this honestly via exit 5.
5. The two-phase flow is implemented in `commands/evaluate.rs` + `commands/execute.rs` CLI-side. The daemon's evaluate/execute handlers do NOT implement `--wait` — they only handle dispatch.

## Notes

**Design decisions:**

- **Router-level branch for subscription methods, not `CommandRequest` variants.** Subscription methods stream notifications outside the request/response envelope, so stuffing them through `cli/dispatch` (which ends with a single `cli/stream-done`) would force either an awkward multi-phase response or a protocol mismatch. Dedicated router branches keep the wire shape clean: a subscription connection is semantically a long-lived bidirectional stream, not a transformed RPC call.

- **Dedicated subscription connections vs multiplexing on one connection.** The PRDv2 data-flow diagrams (lines 475–506) show a separate CLI adapter connection per subscription. Muxing would complicate cancellation (kill one subscription without killing an unrelated in-flight `cli/dispatch`) and would require notification IDs. Dedicated connections are simpler and match the MCP pattern that inspired the daemon architecture.

- **Global semaphore (not per-feature).** Rate-limited providers (Anthropic, OpenAI) enforce account-wide concurrency. A per-feature semaphore lets N concurrent features each open M parallel provider sessions = N·M, which overflows provider limits. The global semaphore keeps the daemon rate-limit-friendly regardless of feature count.

- **`failed-interrupted` recovery fires `FailedInterrupted` events even with no subscribers.** The event-always invariant keeps the bus simple — every state transition emits, always. Subscribers arrive later (dashboard, CI) and replay by reading the manifest + the SQLite audit trail; the live bus is not a history store.

- **Desktop notifications are best-effort and non-load-bearing.** A CI container without `notify-send` or `osascript` would otherwise fail every background completion if notifications were strict. The fallback to terminal bell + stderr line is graceful degradation.

- **No `DecisionSource` trait in Phase 7.** Cycle 1 cut the Phase 6 Task 19 revival; Cycle 2 confirmed the cut. Only `SubscribedGateSource` ships (concrete struct, no trait). If a second consumer shape materializes later (v0.3 dashboard gate UI), extract the trait then — honors the "don't ship scaffolding ahead of a real consumer" rule.

**Trade-offs:**

- **256-event broadcast channel capacity** is a compromise between memory (256 × ~300 bytes × N features) and lag tolerance (slow subscriber drops the tail). A future Phase 7.1 could tune this via `workflow.yaml` if real-world feedback shows lag.

- **8 MiB log buffer per feature** is sufficient for typical 4-minute 7-layer evaluations (observed v0.6 metrics: ~2 MiB total across all layers). Very long-running or verbose layers might overflow — the drop-oldest FIFO keeps memory bounded but may truncate early log history. Accept this for v0.7; revisit if users hit the cap.

- **Connection-close detection via `conn.read_message() -> None`** assumes clean FIN from the client. An abrupt `kill -9` on the CLI leaves the daemon holding a dead connection until its OS-level read returns EOF. In practice this is fast on Unix sockets; Windows named pipes may take up to the configured pipe timeout. Acceptable: subscriptions are lightweight, leaked state is bounded by OS-level connection cleanup.

**Known limitations / follow-ups:**

- **Gate timeout reconciler (Phase 6 Task 8) remains deferred to Phase 6.1.** Phase 7 does NOT introduce the daemon-startup polling reconciler. If a gate times out while nobody runs `pice evaluate`, it stays `Pending` until someone runs evaluate (Phase 6's inline reconcile-on-resume). Phase 7's background mode does NOT accelerate this — the inline reconcile still runs on `--background` dispatch entry.
- **`pice clean` log purge is not wired.** `LogStore::purge` exists but `pice clean` does not call it. Phase 5.5 will wire worktree + log cleanup together.
- **Dashboard gate UI (v0.3) cannot yet consume `gate_requested` notifications** — the handler exists, but no dashboard. Phase 7 emits the events in the correct format; v0.3 is responsible for consuming them.
- **Provider rate limit awareness** — when a provider returns 429, the daemon currently logs + fails the pass. A future release (likely v0.7.x) adds exponential backoff at the job manager level to pause affected features until the rate-limit window passes.
