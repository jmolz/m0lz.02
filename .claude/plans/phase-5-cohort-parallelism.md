# Feature: Phase 5 — Cohort Parallelism (No Worktrees)

> Generated by `/plan-feature` from `PRDv2.md` Phase 5 (lines 2089–2101). Read CLAUDE.md, `.claude/rules/stack-loops.md`, `.claude/rules/daemon.md`, and `.claude/rules/rust-core.md` before implementing.
>
> **Two cycles of adversarial review reduced scope twice.** PRDv2 Phase 5 nominally bundles per-layer execute + 3 merge strategies + worktree isolation + new commands. Cycle 1 (Claude architect + Codex GPT-5.4 xhigh) found that `pice execute` is a single project-root session today (no per-layer write paths), so merge strategies were deferred. Cycle 2 (same reviewers, against the revised plan) verified two implementation-blocking bugs in the worktree slice: (a) the evaluate path uses the `evaluate/create`/`evaluate/score` protocol, NOT `session/create`, so the planned `working_directory` plumbing wires worktrees to nothing the evaluator touches, and (b) `PICE_STUB_LATENCY_MS` was assigned to a Rust file that doesn't exist (the stub provider is TypeScript at `packages/provider-stub/`). Both reviewers' reasoning divergence converged on the same alternative: **ship cohort parallelism only**. The evaluate protocol is already diff-based and context-isolated; the real value of Phase 5 is parallel scheduling, deterministic manifest insertion, and concurrent sink safety. Worktrees move to **Phase 5.5: Worktree Isolation**, whose first task is "extend `EvaluateCreateParams` + every provider impl to carry `working_directory`" — the prerequisite that the worktree slice cannot proceed without.

## Overview

Refactor `stack_loops.rs` so independent layers within a topological cohort run their evaluation passes concurrently via `tokio::JoinSet` + `Semaphore`, capped by `defaults.max_parallelism`. Preserve the determinism invariant (manifest `layers[]` order = DAG topological order, NOT task completion order). Make `PassMetricsSink` thread-safe so a single shared sink can absorb concurrent writes. Add a `PICE_STUB_LATENCY_MS` knob to the TypeScript stub provider so the speedup benchmark can measure parallelism without depending on live API calls.

## User Story

As an AI-assisted developer running a multi-layer evaluation, I want independent layers (e.g., `backend` and `frontend`) to evaluate concurrently so my 7-layer feature completes ≥1.6× faster than sequential, with byte-identical manifest output across runs and no thread-safety regressions in metrics persistence.

## Success Criteria

- [ ] `stack_loops::run_stack_loops` (called from `handlers/evaluate.rs`) runs layers within a cohort in parallel up to `defaults.max_parallelism` (default = `num_cpus::get()`, hard-capped at 16); cross-cohort ordering is unchanged
- [ ] Parallelism is gated: when cohort size = 1, OR when `phases.evaluate.parallel = false` (verify field exists in `EvaluatePhase`; if absent, add with default `true`), the existing sequential path runs unchanged
- [ ] Determinism: manifest `layers[]` ordering is by DAG topological position, NOT by parallel-task completion order. Two back-to-back parallel runs with identical `PICE_STUB_SCORES` produce byte-identical manifest layer order
- [ ] Per-layer context isolation: each parallel task receives only its own layer's contract, filtered diff, and seam findings. Verified by structural inequality on the `EvaluateCreateParams` payload sent to each layer's provider session, NOT by substring matching (Cycle-2 Consider finding)
- [ ] `PassMetricsSink` is `Send + Sync` with `&self` methods (interior mutability inside impls); `Arc<dyn PassMetricsSink>` is shared across all cohort tasks; concurrent `record()` calls do not data-race
- [ ] **Cost telemetry aggregator** (Phase 4.1 surface) audited and made `Send + Sync` for parallel cohort use; concurrent `record_cost()` from N parallel layers produces correct rollup totals
- [ ] **Cancellation**: SIGTERM / SIGINT during a parallel cohort triggers `JoinSet::abort_all()` and shuts down spawned provider sessions before exiting; no orphaned tokio tasks or provider processes survive cancellation
- [ ] `PICE_STUB_LATENCY_MS` env knob added to the **TypeScript** stub provider at `packages/provider-stub/`. When set, every score response sleeps that many ms before returning. Default = 0 (current behavior). **Score consumption is atomic per-layer** (per-layer env vars `PICE_STUB_SCORES_<LAYER>` OR atomic counter on a shared list) so parallel layers don't race on the global score iterator
- [ ] Speedup ≥ 1.6× on the 2-layer reference fixture (`backend` + `frontend`, no `depends_on` edge between them) measured by wall-clock with `PICE_STUB_LATENCY_MS=200` and adaptive `min_passes=1` in the test fixture. Bench assertion: `parallel.mean ≤ 0.625 × sequential.mean`. Bench gating in CI is via a dedicated `#[test]` (not `#[bench]`) that runs the same scenario and `assert!`s the ratio — `cargo bench` alone doesn't fail CI
- [ ] All validation commands pass with zero errors. Rust baseline rises from 811 → projected ~820 with new tests; TS baseline rises from 78 → projected ~81 (stub provider env-knob + atomic-consumption tests)
- [ ] No regressions in existing 811 Rust + 78 TS tests; full Phase 0–4 regression suite green

**`PassMetricsSink` redesign (Cycle-1 finding #5 — `Send + Sync` alone on `&mut self` is meaningless):**

```rust
// Before:
//   pub trait PassMetricsSink {
//       fn record(&mut self, ...);
//   }
// After:
pub trait PassMetricsSink: Send + Sync {
    fn record(&self, evaluation_id: &str, layer: &str, pass: &PassRecord);
}

// NullPassSink: trivially Send + Sync (unit struct, no state)
impl PassMetricsSink for NullPassSink {
    fn record(&self, _: &str, _: &str, _: &PassRecord) {}
}

// SqlitePassSink: wraps Mutex<Connection> internally (already the pattern)
pub struct SqlitePassSink { conn: Mutex<Connection> }
impl PassMetricsSink for SqlitePassSink {
    fn record(&self, eval_id: &str, layer: &str, pass: &PassRecord) {
        let conn = self.conn.lock().expect("sqlite sink poisoned");
        // … existing INSERT logic, unchanged
    }
}
```

The trait method goes from `&mut self` → `&self`; concrete impls move mutable state behind a `Mutex` they already own. Orchestrator holds `Arc<dyn PassMetricsSink>` and clones it into each cohort task. No new struct, no new wrapper.

**Parallel cohort scheduling (replaces the inner `for layer_name in cohort` loop in `stack_loops.rs:~210–240`):**

```rust
let max_parallelism = cfg.workflow.defaults
    .max_parallelism
    .map(|n| n as usize)
    .unwrap_or_else(num_cpus::get)
    .min(16); // hard cap: rate-limit-friendly

let parallel_enabled = cfg.workflow.phases.evaluate.parallel
    && cohort.len() > 1
    && max_parallelism > 1;

if !parallel_enabled {
    // Existing sequential path — unchanged
    for layer_name in cohort {
        let outcome = evaluate_one_layer(/* … */).await;
        manifest.add_layer_result(outcome.into_layer_result());
    }
    continue;
}

// Parallel path
let semaphore = Arc::new(Semaphore::new(max_parallelism));
let sink_arc: Arc<dyn PassMetricsSink> = Arc::clone(&pass_sink);
let mut join_set: JoinSet<(String, LayerOutcome)> = JoinSet::new();

for layer_name in cohort {
    let permit = semaphore.clone().acquire_owned().await?;
    let sink = Arc::clone(&sink_arc);
    let layer = layer_name.clone();
    let layer_inputs = build_per_layer_inputs(&cfg, &layer_name, &manifest_snapshot)?;
    // ^ pass-by-value of LAYER-OWN data only: contract, filtered_diff, seam_checks_for_this_layer.
    //   Never pass `&cfg` or `&manifest` into the spawned task — that's how cross-layer leakage happens.
    join_set.spawn(async move {
        let _permit = permit; // released on drop
        let outcome = evaluate_one_layer(layer_inputs, sink).await;
        (layer, outcome)
    });
}

// Drain — collect into a HashMap so we can reorder by DAG before manifest insert
let mut by_layer: HashMap<String, LayerOutcome> = HashMap::new();
while let Some(joined) = join_set.join_next().await {
    let (l, o) = joined.context("layer task join failed")?;
    by_layer.insert(l, o);
}

// DAG order, NOT completion order — locks the determinism invariant
for layer_name in cohort {
    if let Some(outcome) = by_layer.remove(layer_name) {
        manifest.add_layer_result(outcome.into_layer_result());
    }
}
```

   - Choose primitive: `tokio_util::sync::CancellationToken` (preferred — child tokens propagate cleanly) OR `tokio::sync::watch::Receiver<bool>` if `tokio-util` is not already a transitive dep.
   - Locate the existing daemon SIGTERM/SIGINT handler (likely `pice-daemon/src/main.rs` or `server/lifecycle.rs`); extend it to publish to the cancellation primitive on signal.
   - Pass a child token / receiver clone into each spawned cohort task via the `evaluate_one_layer` signature.
   - JoinSet drain becomes: `tokio::select! { Some(joined) = join_set.join_next() => { … }, _ = cancel_token.cancelled() => { join_set.abort_all(); return Err(StackLoopsError::Cancelled); } }`.
   - Inside `evaluate_one_layer`, the cancellation token is checked at session lifecycle boundaries (before `provider.send_request`, between adaptive passes) so spawned provider sessions can be torn down cleanly via the existing `orchestrator.shutdown()` always-shutdown pattern (`.claude/rules/rust-core.md` → "Session Runner").
   - On cancellation, `handlers/evaluate.rs` returns a typed `ExitJsonStatus::Cancelled` (add the discriminant per the centralize-prefix rule if not already present; if already present, reuse).
5. After `join_set.join_next().await` drains: collect into `HashMap<String, LayerOutcome>`.
6. **Iterate `cohort` (DAG order) to insert into manifest, NOT completion order.** This is the determinism invariant from `.claude/rules/stack-loops.md`.
7. Streaming output: per-layer chunks accumulate into `Vec<String>` inside each task; on JoinSet drain, the orchestrator (in DAG order) flushes them via `sink.send_chunk()`. Avoids interleaved garbage from N concurrent layers writing to one sink. JSON mode is unaffected (sink is no-op).
8. Add `tracing::info!` events at cohort start (cohort_index, layer_count, max_parallelism, parallel_enabled) and cohort end (elapsed_ms, cancelled: bool) — operators see parallelism in action via daemon log. Tracing writes to stderr via `.with_writer(std::io::stderr)` per `.claude/rules/rust-core.md` — does NOT corrupt JSON-mode stdout.

## Notes

### Design decisions

1. **Phase 5 ships parallelism only; worktrees become Phase 5.5.** Two cycles of adversarial review (Tier 3, parallel Claude architect + Codex GPT-5.4 xhigh) converged on this. The codebase reality: `pice evaluate` uses `evaluate/create` / `evaluate/score` (NOT `session/create`); providers do not currently accept a `working_directory` parameter on the evaluate path. Worktrees without that protocol extension are dead code. Phase 5.5 ships the protocol extension first, then the worktree machinery on solid ground.

2. **`max_parallelism` cap of 16 is hardcoded.** Even on a 32-core machine, spawning 32 concurrent provider sessions would swamp Anthropic / OpenAI rate limits. 16 is generous (most reference fixtures have ≤7 layers in any cohort). Users can lower via `defaults.max_parallelism` but cannot raise above the cap. Raising the cap is a one-line change once we have rate-limit-aware backoff in providers (v0.6 concern).

3. **`PassMetricsSink` redesigned, not just bound-widened.** Trait method goes `&mut self → &self`; concrete impls move mutable state behind a `Mutex` they already own (the SQLite impl already does this per `.claude/rules/metrics.md` patterns). Orchestrator holds `Arc<dyn PassMetricsSink>`. Real concurrency design, not a meaningless bound widening.

4. **Per-layer context isolation enforced at the function-signature level.** `build_per_layer_inputs` returns ONLY layer-specific data. Spawned tasks NEVER receive `&cfg` or `&manifest` — context leakage is prevented by the type system, not by hopeful discipline. The structural inequality test (criterion #3) is the runtime backstop, not the primary defense.

5. **Bench uses multi-thread runtime, NOT `tokio::time::pause()`.** Pausing tokio's clock would silently zero out the stub's `setTimeout`, producing a meaningless bench. Top-of-file comment in `parallel_cohort_speedup.rs` documents the requirement.

6. **No new schema fields, no new RPC variants, no new ExitJsonStatus discriminants, no new config files.** Phase 5 reuses pre-existing `defaults.max_parallelism` and `phases.evaluate.parallel` (the latter verified or added in Task 1). The only schema delta is potentially adding `EvaluatePhase.parallel: bool` if it's not already there. Minimum surface area.

7. **Per-feature serialization is unchanged.** The Pass-8 manifest file lock in `handlers/evaluate.rs` continues to serialize same-feature evaluations. Phase 5 introduces no new lock primitive. Cross-feature parallelism is out of scope (Phase 7: Background Execution).

8. **Cancellation primitive: `tokio_util::sync::CancellationToken` preferred over `tokio::sync::watch::Receiver<bool>`.** CancellationToken supports child tokens that propagate cancellation cleanly through the cohort task tree; the `watch` channel would need manual cascade. If `tokio-util` is not already a transitive dep, accept the small dep addition (it's already used by `tokio` itself for several feature flags, and `tokio-util = { version = "0.7", features = ["rt"] }` is the minimum import). Verify in Task 1 alongside `num_cpus`; if `tokio-util` is absent, the dep addition is also documented in Task 1.

9. **Cost aggregator gets the same redesign as `PassMetricsSink`, not a separate scheme.** Both surfaces have the same fundamental requirement (concurrent `record_*` calls from N parallel cohort tasks); both use the same pattern (`&self` + `Send + Sync` + interior mutability inside concrete impls + `Arc<dyn>` at orchestrator). Symmetry keeps the codebase coherent. If Phase 4.1's cost aggregator turns out to use a fundamentally different pattern (e.g., channel-based), revisit and document the divergence — but the default assumption is symmetry.

10. **Bench has a companion CI-gating test.** `cargo bench` does not fail CI on regression — criterion only reports. The dedicated `#[tokio::test(flavor = "multi_thread")]` at `tests/parallel_cohort_speedup_assertion.rs` is what gates: it runs the same scenario with N=3 (bench uses N=10 for stability), measures, and `assert!`s the ratio. Two files, one source of truth (the test fixture lives in a shared module imported by both).

### Known limitations

- **Stub provider latency simulation is jitter-prone on shared CI runners.** The `≤ 0.625× sequential` bench assertion has a small but non-zero risk of flake under heavy CI contention. If CI flakes during execution, the remediation is: switch the bench to running on a dedicated job with `runs-on: ubuntu-latest-4-cores` (or similar) AND increase sample size; do NOT loosen the assertion.
- **Single-thread tokio runtime is unsupported.** Phase 5 assumes multi-thread runtime. The CLI's `#[tokio::main]` already uses multi-thread by default; the bench explicitly opts in. If a future change switches to single-thread, parallelism becomes a no-op AND the bench will fail (correctly catching the regression).
