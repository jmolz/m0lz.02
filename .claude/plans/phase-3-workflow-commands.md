# Feature: Phase 3 — Workflow Commands

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Implement the 5 remaining workflow commands (`pice prime`, `pice review`, `pice commit`, `pice handoff`, `pice status`) and a reusable session runner, completing the full PICE workflow lifecycle so a developer can run the entire cycle from orientation through handoff using only CLI commands.

## User Story

As a developer using the PICE methodology, I want to run `pice prime` through `pice commit` and `pice handoff` using CLI commands, so that I have a complete structured workflow from codebase orientation to session handoff without manually invoking AI sessions.

### Task 1: Reusable Session Runner Module

**File:** `crates/pice-cli/src/engine/session.rs`
**Action:** CREATE
**Description:** Extract the duplicated session lifecycle pattern from plan.rs and execute.rs into a reusable module. This module provides two functions:

1. `run_session()` — Creates session, sends prompt, destroys session. For commands that only need streaming output (prime, review).
2. `run_session_and_capture()` — Same lifecycle but also captures all `response/chunk` text into a String. For commands that need the AI response text (commit, handoff).

Both functions take a `&mut ProviderOrchestrator`, project root path, and prompt string. The capture variant uses `Arc<Mutex<Vec<String>>>` to collect chunks from the notification handler, similar to the oneshot channel pattern in evaluate.rs.

```rust
// engine/session.rs — key signatures:
pub async fn run_session(
    orchestrator: &mut ProviderOrchestrator,
    project_root: &Path,
    prompt: String,
) -> Result<()>

pub async fn run_session_and_capture(
    orchestrator: &mut ProviderOrchestrator,
    project_root: &Path,
    prompt: String,
    print_chunks: bool,
) -> Result<String>
```

The `print_chunks` parameter controls whether chunks are also printed to stdout via `output::print_chunk()` while being captured. This is useful for `handoff` (print + capture) vs `commit --json` (capture only).

**Depends on:** none
**Validate:** `cargo check -p pice-cli`

All functions return `Result<String>` and handle missing git repos gracefully (return empty string, don't panic).

### Task 9: Implement `pice commit` Command

**File:** `crates/pice-cli/src/commands/commit.rs`
**Action:** MODIFY
**Description:** Replace the stub with a full implementation. This command is unique: it uses the provider to generate a commit message, then executes `git commit` locally.

CLI args:
```rust
pub struct CommitArgs {
    /// Override commit message (skip AI generation)
    #[arg(short, long)]
    pub message: Option<String>,

    /// Show generated message without committing
    #[arg(long)]
    pub dry_run: bool,

    /// Output as JSON
    #[arg(long)]
    pub json: bool,
}
```

Implementation:
1. Load config
2. Check git status — fail early if nothing to commit (`git status --porcelain` is empty)
3. **If `--message` provided**: skip provider, use the manual message directly
4. **Otherwise**: build commit prompt via `prompt::build_commit_prompt(&project_root)`, start provider, use `session::run_session_and_capture()` to get the generated message, shutdown provider
5. Clean the captured message: trim whitespace, strip markdown code fences if present (providers sometimes wrap in ```), remove trailing newlines
6. **If `--dry-run`**: print the message and exit (don't commit)
7. **Otherwise**: stage tracked changes (`git add -u` if nothing explicitly staged), then run `git commit -m "<message>"` via `std::process::Command`
8. Output: `{"status": "committed", "message": "<message>"}` for JSON, or print the message + "Committed." for text
9. Exit code: 0 if commit succeeded, 1 if git commit failed

**Depends on:** Task 1, Task 4
**Validate:** `cargo check -p pice-cli`

**Rules for writing criteria:**

- Each criterion must be independently testable — no "works well" or "looks good"
- Include the validation method: a command to run, an API call to make, or an observable behavior
- Set higher thresholds (8-9) for security, data integrity, and multi-tenancy concerns
- Set standard thresholds (7) for functionality and UX
- Include at least one negative criterion (e.g., "Returns 403 when accessing another org's data")

## Notes

- **No new protocol methods needed.** Phase 3 commands use the existing `session/create`, `session/send`, `session/destroy` methods. They differ only in prompts and post-processing.
- **No stub provider changes needed.** The stub already handles all session/* methods. Phase 3 commands just send different prompts.
- **Metrics collection deferred to Phase 4.** Phase 3 commands do NOT write to metrics.db — that infrastructure doesn't exist yet and is explicitly Phase 4 scope.
- **State machine deferred.** `.pice/state.json` persistence is mentioned in `rust-core.md` but is not implemented in Phase 2. Phase 3 `pice status` derives state from filesystem scanning (plan files) and git status rather than a state file. A formal state machine can be added in Phase 4 when metrics collection provides a reason to track state transitions.
- **`pice commit` auto-stages tracked changes.** If nothing is explicitly staged when `pice commit` runs, it stages all tracked modified files (`git add -u`). It does NOT stage untracked files — the user must `git add` new files manually.
- **Prompt quality matters for real providers but not for testing.** The stub provider echoes regardless of prompt content. Tests verify the pipeline (config → provider → session → output), not prompt quality. Prompt refinement happens when testing with real Claude Code provider.
