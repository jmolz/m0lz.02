# Feature: Spec Traceability During Execution

> Generated by the migrated `plan-feature` command. Read `AGENTS.md`, `.codex/rules/rust-core.md`, `.codex/rules/daemon.md`, and `.codex/rules/templates.md` before implementing.

## Overview

Luca's question exposed a product clarity and runtime enforcement gap: PICE already ties implementation back to a plan and contract, but the answer should not be "prime ties back to the PRD." `prime` orients. The binding work happens when `plan` converts the original request/spec into an approved plan plus contract, `execute` starts a fresh session from that plan, and `evaluate` grades the resulting diff against the contract.

This feature makes that chain explicit and mechanically harder to bypass. It adds first-class spec traceability guidance to generated plans and public docs, strengthens the execution prompt, makes `pice execute` fail closed when the plan lacks a parseable `## Contract`, persists plan/contract hashes in manifests, and ensures Stack Loops can fall back to the approved feature contract instead of a generic layer contract when layer-specific contracts are absent.

## User Story

As a PICE user explaining the workflow to another developer, I want the implementation phase to visibly and mechanically stay tied to the original spec through the approved plan and contract, so that agent work cannot drift into untraceable implementation once coding starts.

## Success Criteria

- [ ] Generated plan templates and `pice plan` prompts require a spec traceability section with explicit fields for mapping original inputs to plan tasks and contract criteria.
- [ ] Spec lineage is explicitly bounded to material already supplied in the planning prompt or manually cited stable references; this feature does not discover PRDs, issues, emails, or external source systems automatically.
- [ ] `pice execute` foreground mode refuses to start a provider session when the plan has no parseable `## Contract`.
- [ ] `pice execute --background` performs the same contract preflight before writing a queued manifest or dispatching a background job, while preserving the current inline-mode rejection precedence.
- [ ] Background execute/evaluate manifests persist enough trace metadata to prove which plan and contract were used: plan path, plan hash, contract hash, contract feature, contract tier, and whether the plan carried a `## Spec Traceability` section.
- [ ] Stack Loops layer evaluation falls back to the approved feature contract when a layer-specific contract file is absent, not a generic placeholder.
- [ ] `pice status` list mode discovers Codex-primary plans in `.codex/plans` as well as Claude-primary plans in `.claude/plans`.
- [ ] The execution prompt explicitly says the approved plan and contract are the source of truth and that deviations must be surfaced instead of silently changing scope.
- [ ] Public README and methodology docs answer Luca's question accurately: `prime` orients, but implementation is tied back through plan content, contract criteria, isolated execution, and contract evaluation.
- [ ] Existing evaluate contract behavior remains unchanged.
- [ ] All validation commands pass.
- [ ] No regressions in existing behavior.

### Patterns to Follow

Naming:

```text

Structured failure status strings: use ExitJsonStatus::as_str(), never raw literals

```

Error handling:

```text
Foreground text failures use CommandResponse::Exit.
JSON-mode failures use CommandResponse::ExitJson.
Malformed plans use plan-parse-failed exit family.
Evaluate no-contract plans keep ExitJsonStatus::NoContractSection and exit 2 because evaluation cannot grade the contract.
Execute no-contract plans use a command-precondition failure status, `plan-contract-required`, and exit 1.
```

## Current State

- `build_plan_prompt` asks providers to include a `## Contract` section with JSON criteria and writes plans into `.codex/plans` or `.claude/plans` based on provider name.
- `build_execute_prompt` includes the full plan content and tells the provider to execute tasks in order and run validation after each task.
- `pice execute` foreground mode calls `ParsedPlan::load`, builds the prompt from `plan.content`, and starts the provider. It does not check `plan.contract` before execution.
- `pice execute --background` snapshots raw plan file content at dispatch time and starts a background job. It does not require the contract before writing the queued manifest.
- `pice evaluate` already fails closed when a plan has no `## Contract`, returning `ExitJsonStatus::NoContractSection` in JSON mode and exit code 2.
- `pice execute` should not reuse evaluate's exit-2 contract-failure family; a contract-free execute request is a command precondition failure and should exit 1.
- Verification manifests are keyed by feature id derived from the plan filename stem and store layer/gate/status/run data, but not plan path, plan hash, contract hash, or source/spec references.
- Stack Loops parses the top-level plan contract before evaluation, but per-layer scoring uses layer TOML contracts. When a layer contract is missing, the current fallback is a generic placeholder rather than the approved feature contract.
- `pice status` list mode scans `.claude/plans` only, so Codex-primary plans in `.codex/plans` can be invisible in the historical plan list.
- Docs already describe the intended model: implementation gets a fresh session with plan content only, and evaluators see contract plus diff plus `AGENTS.md`.
- Docs overstate current execute enforcement by saying execute extracts the contract to validate that the file is a proper PICE plan.
