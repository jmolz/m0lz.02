> Generated by the migrated `plan-feature` command for: `PRDv2.md Phase 8`.
> Read `AGENTS.md`, `PRDv2.md`, and referenced rules before implementing.

## Overview

PRDv2 Phase 8 ships the completed v0.2 Stack Loops line as a release-ready product. The current checkout already has Phase 7 background execution merged into `main`, but the release surface is not yet coherent enough to tag:

- public docs still describe the old single-binary, v0.1-centered command surface in several places;
- `README.md` and `CONTRIBUTING.md` test counts are stale relative to the repo's expected Phase 7+ baseline;
- `README.md` does not yet prove that test counts, benchmark claims, screenshots, diagrams, images, and GIFs were refreshed from newly run evidence;
- package manifests are at `0.6.0`, PRDv2 Phase 8 says `v0.2.0`, and the Phase 7 plan refers to `v0.7.0`;
- npm platform packages copy `pice-daemon` during release but their `"files"` arrays omit the daemon binary, while Rust auto-start still runs `Command::new("pice-daemon")`;
- CI does not yet prove the full Phase 8 acceptance suite across macOS arm64/x64, Linux arm64/x64, and Windows x64;
- there is no durable `fixtures/reference-projects/` suite for the five PRDv2 reference framework projects;
- `.codex` project guidance and `.claude` product scaffolding are inconsistent and must be reconciled before public release wording is frozen.

This phase produces the release artifact, not a new core orchestration feature. Implementation should therefore prefer small, mechanically verifiable changes: docs aligned to code as written, CI/release gates that exercise shipped commands, packaging fixes, reference fixtures, and release notes.

## User Story

As a developer adopting PICE v0.2, I want the published CLI, npm package, docs, migration flow, and cross-platform release artifacts to match the Stack Loops behavior that actually shipped, so that I can upgrade from v0.1, run layer-aware/background evaluations, and trust that release claims were verified on my platform.

### Patterns to Follow

**Docs must be mechanically tied to code.**

- Before changing command docs, run `cargo run -p pice-cli -- --help` and subcommand help for `init`, `layers`, `validate`, `evaluate`, `status`, `logs`, `review-gate`, and `daemon`.
- Do not copy PRDv2 claims into docs unless current code or release tests prove them.
- If a feature is implemented but has a known limitation, state it as a limitation, not a shipped guarantee.

**Release tests should prove artifacts, not just source builds.**

- Source-level `cargo test` is necessary but insufficient for Phase 8.
- Unpack generated release assets and run the binaries from the artifact directory.
- npm release tests must install from packed local tarballs and verify the wrapper can locate both `pice` and `pice-daemon`.

**README evidence must be current and reproducible.**

- Do not update README test counts, benchmark claims, badges, GIF captions, or diagrams from memory or old CI output.
- Every README benchmark number must name the command, date, environment/runner, and whether it is an informational Criterion benchmark or a pass/fail regression assertion.
- Every README image/GIF reference must resolve to a checked-in asset or intentionally external URL, include useful alt text, and match the v0.2 daemon/background/review-gate command surface.
- If a demo GIF or diagram cannot be regenerated during Phase 8, either label it historical or remove it from first-party release claims.

**Do not silently rename `.claude` to `.codex`.**

- Current code scaffolds `.claude/`.
- Current repo-local Codex guidance uses `.codex/`.
- Phase 8 must make one explicit product decision:
  - keep `.claude` as the public scaffold for compatibility and update `.codex/rules/templates.md` wording to say root `.codex` is repo-local only; or
  - implement a compatibility migration to `.codex` with `.claude` fallback.
- Do not mix the two in public docs without explaining the compatibility story.

**Packaging must include the daemon.**

- `npm/pice/index.js` already has `getDaemonBinaryPath()`.
- `npm/pice*/package.json` currently includes only the CLI binary in `"files"`; update platform packages to include `pice-daemon` / `pice-daemon.exe`.
- Rust auto-start currently calls `Command::new("pice-daemon")`; either ensure npm installation exposes daemon on PATH, or teach the CLI to resolve the bundled daemon path through a stable env/config handoff from the npm wrapper.

## Adversarial Review

**Tier**: 3
**Reviewers**: GPT-5.5 architect + GPT-5.5 code-reviewer, two independent cycles
**Refinement cycles**: 2
**Attack framework**: Karpathy four principles + PoetiQ cross-model verification

### Critical

- Resolved: release version/tag/package identity cannot be left to the executor. Task 0 now requires an explicit approval block before version bumps, docs publication wording, tag, or publish.
- Resolved: public metrics/telemetry docs originally depended on PRDv2 claims instead of live schema evidence. Task 2 now inventories a temp initialized project before docs can cite v0.2 metric surfaces.
- Resolved: version bump originally happened after validation. Task 13 now applies the approved version before Task 14 final validation.
- Resolved: acceptance/release gates originally used POSIX shell selectors and would not close the Windows/platform obligation. Task 5 and Task 7 now define canonical Node harnesses.
- Resolved: review-gate acceptance was under-specified. Task 5 now requires deterministic `review.enabled: true`, an always-triggered gate, expected exit code `3` capture, explicit `--gate-id` approval, and resume evidence.
- Resolved: `status --follow` / `logs --follow` smoke could hang. Task 5 now uses exact feature-scoped stream-json commands with child-process timeouts and terminal-frame assertions.
- Resolved: release publishing could swallow npm failures. Task 7 now requires fail-closed publish handling except for verified already-published package versions.

### Consider

- Reference fixtures are intentionally framework-shaped and offline; they prove PICE release behavior, not that generated framework apps build end to end.
- CI architecture coverage may need documented gaps for runner availability, but release notes must state any gap explicitly rather than implying full native execution.
- `.claude` vs `.codex` remains a product decision, not a planning decision. Task 0 requires the user-approved namespace policy before implementation freezes docs/templates.
- Release notes should stay under `docs/releases/UNRELEASED.md` until the version/tag decision is approved.
- README benchmark and media requirements were added during final contract approval; implementation should treat them as first-class release blockers even though they were added after the two adversarial cycles.

### Reasoning Divergence

- One review line treated Phase 8 mostly as documentation hardening; the other treated it as release engineering. The final plan keeps docs work blocked behind release identity, schema evidence, packaging, and artifact smoke gates.
- Reviewers disagreed on whether to implement missing metrics schema during Phase 8 or call it a limitation. The final contract allows either path, but only with explicit release-readiness matrix evidence and user signoff for limitations.
- Both reviewers converged that tag/publish is a separate approval boundary because current release workflow publishes GitHub and npm artifacts on tag push.

## Notes

- This plan intentionally treats "release" as a gated operation. Implementation may prepare version bumps and release notes, but tagging/publishing requires a separate explicit approval.
- If the team decides Phase 8 should publish as `0.7.0` rather than `0.2.0`, rename release-note paths and contract wording consistently. Do not leave both release names in user-facing docs.
- If `.claude` remains the public scaffold, update repo-local `.codex` rule wording to avoid future agents "fixing" code by renaming the product namespace accidentally.
- Do not publish README benchmark numbers without command/date/environment context. Prefer a compact evidence table over decontextualized performance claims.
