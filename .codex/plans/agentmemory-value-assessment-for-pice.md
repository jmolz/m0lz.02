## Overview

This plan answers whether `rohitg00/agentmemory` contains value PICE can use. It does **not** implement runtime session memory, prompt injection, replay/import, new config, new SQLite tables, or provider lifecycle changes.

The deliverable is a research/RFC document that records the inspected AgentMemory snapshot, the reusable patterns, the rejected integration paths, and a future PICE-native implementation boundary. This revision folds in the first adversarial review by narrowing scope to the original request and by moving runtime design decisions into documented recommendations for a separate follow-up plan.

Preliminary conclusion to verify in the RFC: AgentMemory has strong value as prior art for durable agent memory, token-budgeted recall, safe replay/import, commit/session linking, and governance. It should not be adopted as a direct PICE runtime dependency because PICE already owns workflow lifecycle, manifests, metrics/audit, provider JSON-RPC, daemon RPC, and evaluator isolation.

## User Story

As a PICE maintainer, I want a source-backed assessment of AgentMemory's reusable value and integration risks, so that we can decide what to adapt into PICE without silently introducing a second memory runtime or weakening evaluator isolation.

## Adversarial Review

**Tier**: 1 after scope reduction to documentation/RFC only
**Reviewers**: local adversarial pass against the saved plan, `AGENTS.md`, recent git history, and targeted source checks. Parallel subagents were not spawned because the current Codex instructions require explicit user authorization before delegation.
**Refinement cycles**: 2
**Attack framework**: Karpathy four principles + PoetiQ cross-model verification

### Cycle 1 Critical Findings And Revisions

1. Hidden assumption: the original draft converted "determine whether there is value" into a full runtime implementation. Resolution: scope is now assessment/RFC only; runtime memory is a future plan.
2. Verification-loop readiness: the original draft assumed `run_session()` could persist memory without defining config, command, feature, run, or metrics handles. Resolution: no session code changes in this plan; the RFC must state that future work needs a `SessionMemoryRecorder` or `SessionRunContext`.
3. Hidden architecture coupling: the original draft put retrieval in prompt builders that should not own storage/search policy. Resolution: no prompt builder changes in this plan; the RFC must state that future prompt builders receive precomputed context instead of opening storage.
4. Evaluation-adjacent contamination risk: the original draft allowed memory in `review` and possibly `commit`. Resolution: the revised plan requires the RFC to exclude `review`, `evaluate`, adversarial review, and `commit` from any first runtime memory pass unless separately approved.
5. Weak success criteria: replay/import was optional in prose but present in the contract. Resolution: replay/import is now future scope only, documented as needing a separate contract.
6. Privacy hidden assumption: the original draft suggested raw prompt capture once memory was enabled. Resolution: the revised plan rejects raw prompt/tool capture by default and requires explicit future approval.
7. Storage source-of-truth ambiguity: the original draft assumed `.pice/metrics.db` was the right store. Resolution: the assessment must compare `.pice/learnings.md`, `.pice/metrics.db`, and `~/.pice/state` before recommending a future implementation.

### Cycle 2 Critical Findings

None requiring plan changes before execution.

### Cycle 2 Consider

1. The RFC should avoid overstating AgentMemory benchmark or token-savings claims unless it cites them as project claims rather than independently verified facts.
2. The scope guard now unions `git diff --name-only` with `git ls-files --others --exclude-standard` so newly created docs are covered.
3. The cross-link in `docs/research/self-evolving-verification.md` should be short. The assessment document should carry the detail.

### Cycle 2 Reasoning Divergence

1. Alternative: approve runtime memory implementation now. Rejected for this plan because the user's original request was value assessment, and the first review showed runtime integration needs its own design contract.
2. Alternative: skip the cross-link and create only the RFC. Rejected because existing self-evolving verification research is the natural place future implementers will look for PICE memory strategy.
3. Alternative: make the future recommendation `.pice/learnings.md` only. Rejected as premature; the assessment should compare options and recommend a likely starting point without binding the later implementation plan.
