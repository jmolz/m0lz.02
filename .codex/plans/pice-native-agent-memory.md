# Feature: PICE-Native Agent Memory

> Generated by the migrated `plan-feature` command. Read `AGENTS.md`, `PRDv2.md`, `.codex/rules/daemon.md`, `.codex/rules/stack-loops.md`, `.codex/rules/metrics.md`, and `.codex/rules/protocol.md` before implementing.

## Overview

This feature adds an opt-in, PICE-owned memory layer for durable project lessons and bounded workflow recall. It implements the follow-up recommended by the AgentMemory value assessment without adopting AgentMemory as a runtime dependency, MCP server, provider-side tool, hook lifecycle, vector store, graph search system, or raw transcript archive.

The first implementation is deliberately narrow where it affects model context and comprehensive where it affects governance. PICE may write summary-only memory records after approved daemon lifecycle points, assemble token-budgeted memory briefs for approved workflow prompts, expose operator-visible memory inspection and prune commands, and record memory access metadata in metrics. PICE must not feed memory into `evaluate`, adversarial evaluation, `review`, or `commit`, and memory must never become the source of truth for manifests, gates, contracts, evaluations, or release readiness.

## User Story

As a PICE user working across repeated repository sessions, I want PICE to remember durable project lessons and recurring workflow gotchas only when I opt in, so that `prime`, `plan`, and `execute` can start with relevant context while evaluator, review, and commit isolation remains mechanically provable.

## Success Criteria

- [ ] `.pice/config.toml` supports an opt-in `[memory]` section with conservative defaults and old configs continue to parse.
- [ ] PICE adds no AgentMemory runtime dependency, MCP wiring, provider-side memory call, embeddings package, vector database, graph index, or new hook lifecycle.
- [ ] Memory policy is centralized in Rust core types and denies `review`, `evaluate`, adversarial evaluation, and `commit` regardless of handler behavior.
- [ ] `prime`, `plan`, and `execute` can receive a bounded precomputed memory brief only when memory is enabled and relevant records exist.
- [ ] `handoff` and successful workflow phases can write summary-only records through a daemon-owned `SessionMemoryRecorder`; no raw prompts, raw diffs, raw tool output, or provider transcripts are persisted by default.
- [ ] Prompt builders stay storage-free: handlers pass `Option<MemoryBrief>` or equivalent precomputed context; builders do not open SQLite, read `~/.pice/state`, run retrieval, or call tools.
- [ ] Metrics records memory write/read/prune metadata in SQLite without storing natural-language memory content as metrics truth.
- [ ] `pice memory` exposes status, list, show, and prune/delete operations with `--json` parity and stable exit codes.
- [ ] Memory storage is project-scoped by normalized project root and project hash; private state under `~/.pice/state/{project_hash}` cannot collide across repositories.
- [ ] Retention, pruning, and redaction are implemented in the first runtime version, not deferred.
- [ ] Explicit tests prove no memory text reaches `evaluate/create`, adversarial evaluator prompts, `pice review`, or `pice commit`.
- [ ] Documentation explains memory as advisory context, not authoritative workflow state.
- [ ] All validation commands pass.
- [ ] No regressions in existing behavior.

### Patterns to Follow

Naming:

```text

Daemon module: crates/pice-daemon/src/memory
```

Error handling:

```text
Memory disabled is not an error.
Missing memory store is not an error for read paths; return an empty brief and record no read event unless configured to audit misses.
Corrupt private-state memory files are a command failure for pice memory show/list and a warning plus empty brief for workflow prompts.
Redaction failure prevents a memory write and records memory_write_rejected metadata.
Memory read/write failure must not cause evaluation, review, or commit behavior because those handlers must not call memory.
JSON command failures use CommandResponse::ExitJson with stable status strings and exit code 1.
```

Imports:

```text
Rust uses explicit imports.
Core memory types live in pice-core; daemon storage and filesystem logic live in pice-daemon.
Prompt builders receive precomputed memory brief values and never import storage modules.
Provider protocol crates do not import memory modules for this feature.
```

## Current State

- PICE has no first-class memory config, command, storage abstraction, recall policy, retention policy, or memory metrics.
- `.pice/config.toml` currently covers provider, evaluation, telemetry, metrics, and init settings; adding `[memory]` must be backward-compatible for existing configs.
- Workflow prompts are assembled in daemon prompt builders. `prime`, `plan`, and `execute` are the only first-pass commands allowed to receive recalled memory.
- `handoff` and `commit` use `run_session_and_capture()`. `handoff` can be a write source after redaction; `commit` must remain staged-diff scoped and must not read or write memory.
- `review` uses workflow guidance plus git diff/status. It must remain memory-free.
- `evaluate` uses contract, diff, and evaluation guidance from `AGENTS.md`; Stack Loops layer evaluation must remain memory-free and context-isolated.
- Metrics SQLite schema currently stores evaluation, criteria, loop events, telemetry, pass events, seam findings, gate decisions, and layer runs. It does not store memory events.
- The verification manifest remains the source of truth for feature/layer/gate/evaluation state. Memory must not modify manifest semantics except possibly recording advisory memory links in metadata only if a later task proves that does not create authority confusion.
- The AgentMemory assessment already rejects direct runtime dependency, AgentMemory server/MCP source of truth, provider-side memory calls, hook-driven lifecycle, raw prompt/tool capture by default, evaluator/review prompt injection, and immediate vector/graph/rerank implementation.

## Adversarial Review

**Tier**: 3
**Reviewers**: Primary planner plus local adversarial pass against `AGENTS.md`, the AgentMemory assessment, daemon rules, Stack Loops rules, metrics rules, and protocol rules.
**Refinement cycles**: 1

### Critical

1. The plan could be misread as "implement AgentMemory inside PICE" instead of "learn from AgentMemory patterns." Resolution: the overview, success criteria, Task 11, and contract explicitly forbid AgentMemory runtime dependency, MCP wiring, provider calls, embeddings, vector search, graph search, reranking, and hook lifecycle adoption.
2. Memory could contaminate evaluator context through shared prompt-builder plumbing. Resolution: memory policy is default-deny, prompt builders accept explicit optional briefs only for approved commands, and Task 8 requires sentinel tests against review, commit, evaluate, adversarial evaluation, Stack Loops layer payloads, and adaptive payloads.
3. Memory could become a second source of truth for workflow state. Resolution: the plan treats memory as advisory context only, keeps manifests authoritative, and limits metrics to metadata without natural-language memory content.
4. Summary memory could still leak secrets or private incident data. Resolution: Tasks 3, 4, 9, and 10 require redaction, rejection, inspection, prune/delete, retention policy, and documentation in the first runtime version.
5. Prompt builders might quietly grow filesystem or SQLite access. Resolution: Task 7 requires storage-free builder signatures and Task 8/contract include negative searches and tests.
6. `execute` streamed output is not naturally available for high-quality memory summaries. Resolution: Task 3 permits only deterministic run summaries for `execute` in v1, while `handoff` may use already captured output. Raw streamed transcript capture is explicitly out of scope.
7. Config could silently accept unsafe future keys. Resolution: Task 1 requires explicit handling or tests for unsupported memory-like keys based on the repository's current serde behavior.

### Consider

1. `.pice/learnings.md` is transparent but commit-prone. The plan allows `private_state` and `both`; implementation should make docs clear enough that users choose shared vs private memory intentionally.
2. A `pice memory add` operator-note command may be useful, but it is not required for v1. If added, it must use the same redaction and metrics path as recorder writes.
3. Retention for committed `.pice/learnings.md` is socially different from retention for private JSONL state. The implementation should avoid pretending that pruning a committed file erases prior git history.
4. Memory summaries can become stale. The brief renderer should show created timestamps/source phase so users and models can treat them as historical context.
5. If schema version numbers have advanced by implementation time, Task 6 must use the next actual schema version, not the number implied by this plan.

### Reasoning Divergence

1. Direct AgentMemory dependency - why not: it would add an external runtime, state model, hooks, MCP/server surface, dependency cadence, and possible lifecycle authority outside the PICE daemon.
2. Metrics-only natural-language memory - why not: it would blur analytics/audit with prompt context and make SQLite look authoritative for memory content.
3. Provider-side memory calls - why not: providers would bypass daemon policy, redaction, metrics, and context-isolation tests.
4. Memory everywhere - why not: evaluator, review, and commit prompts have stronger isolation requirements than workflow prompts.
5. Vector/graph/rerank first - why not: the current value is durable bounded lessons, not retrieval sophistication; retrieval complexity should follow measured need and a separate contract.
6. Raw transcript replay/import first - why not: import/replay carries high privacy and provenance risk and needs its own allowlist, symlink, malformed-line, dry-run, and delete contract.

## Notes

- This plan is an implementation plan for PICE-native summary memory v1. It is not an implementation of AgentMemory.
- The feature should be executed in a worktree. Do not edit `main` directly for implementation.
- If implementation discovers that some handlers cannot be tested without refactoring prompt assembly, prefer a small dependency-injection seam over weakening the no-leak tests.
- The plan intentionally starts with deterministic recall. Any model-generated summary or semantic search path must be introduced behind a later contract and evaluated for contamination risk.
- A future memory v2 can consider commit/session linking, import/replay, vector recall, graph recall, reranking, and self-evolving rule suggestions after v1 proves governance and isolation.
