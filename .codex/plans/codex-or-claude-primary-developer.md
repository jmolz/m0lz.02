# Feature: Codex Or Claude Primary Developer

> Generated by the migrated `plan-feature` command. Read `AGENTS.md`, `.codex/rules/providers.md`, `.codex/rules/protocol.md`, and `.codex/rules/daemon.md` before implementing.

## Overview

PICE currently treats Claude Code as the only first-class workflow developer provider. The `claude-code` provider declares `workflow: true` and implements `session/create`, `session/send`, and `session/destroy`; the `codex` provider is registered but declares `workflow: false` and only implements evaluation. Users can configure Codex as an adversarial evaluator, but they cannot use Codex as the primary developer behind `pice prime`, `pice plan`, `pice execute`, `pice review`, `pice commit`, or `pice handoff`.

This feature has two explicit lanes. The runtime lane makes the primary developer provider genuinely selectable between Claude Code and Codex. The process-surface lane adds a public Codex scaffold, including `self-heal`, because Codex-primary users need Codex-native command files and project guidance. The self-heal step is post-merge hygiene: it should harvest durable lessons after the feature worktree has landed on `main`, not mutate process guidance while a feature branch is still unaccepted.

## User Story

As a PICE user, I want to choose either Claude Code or Codex as the primary developer behind PICE workflow commands, so that I can run the PICE lifecycle with the AI coding tool and auth model I already use.

## Adversarial Review

**Tier**: 2
**Reviewers**: isolated architect reviewer + isolated GPT-5.5 xhigh code-reviewer fallback
**Refinement cycles**: 1

### Critical (resolved before proceeding)

1. Codex CLI contract was undefined - section: Task 3 - resolution: added a prerequisite task pinning `codex exec --json --cd <workingDirectory> --sandbox workspace-write --output-last-message <temp-file> -`, minimum `codex-cli 0.130.0`, required flag checks, optional `--ask-for-approval never` when advertised by the installed CLI, and unsupported-version errors.
2. Provider-neutral guidance risked leaking `CLAUDE.md` into evaluator prompts - sections: Task 1 / Task 2 - resolution: split workflow guidance from evaluation guidance; evaluator guidance remains AGENTS-only and gains tests proving `CLAUDE.md` exclusion.
3. Commit/handoff capture semantics were underspecified - sections: Task 4 / Task 7 - resolution: final text is captured from `--output-last-message`; `run_session_and_capture()` prefers `response/complete.result.finalText`, and tests assert `commit --dry-run --json` plus `handoff --json` captured output without concatenating divergent streamed progress.
4. Fake fixtures could pass while real Codex CLI drifted - section: E2E Validation - resolution: added non-secret Codex CLI contract smoke for version and required flags before live model calls.
5. Scaffolding and self-heal scope was broader than runtime provider selection - sections: Task 8 / Task 12 - resolution: made scaffold/self-heal an explicit second lane, removed `--developer both`, and added contract criteria for Codex scaffold/self-heal.

### Consider

1. Direct Responses API workflow provider could be more stable than Codex CLI JSONL. Not chosen for this plan because user-facing goal is Codex as a primary developer, and CLI-backed workflow preserves Codex CLI auth/session expectations. The plan now contains a CLI contract smoke to limit this risk.
2. Keyword-only doc validation was weak. Resolution: kept grep as inventory but added init/config parse tests where docs/templates can be mechanically checked.
3. Cross-platform fixture risk is real given recent Windows smoke history. Resolution: fixture must be a Node script, not POSIX shell.
