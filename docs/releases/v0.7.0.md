# PICE v0.7.0 Release Notes

Status: draft release notes for the Phase 8 release-readiness branch. Do not tag
or publish from this branch until the final checklist is approved.

## Release Identity

- Release tag: `v0.7.0`
- Cargo, TypeScript package, and npm package version: `0.7.0`
- Public scaffold namespace: `.claude/` remains the user-facing scaffold emitted by `pice init`
- Maintainer namespace: `.codex/` is repo-local command, rules, and planning context
- Publish boundary: GitHub tag/release approval does not approve npm publish; npm publish requires a separate manual workflow approval after dry-run evidence

## Highlights

- Stack Loops layer detection, dependency cascade, always-run layer handling, seam checks, adaptive evaluation, and review gates are the v0.2 product surface.
- `pice-daemon` is the long-running orchestrator. The `pice` CLI is an adapter for command parsing, terminal output, daemon lifecycle, and socket dispatch.
- Background evaluation supports `pice evaluate --background --wait`, `pice status --follow --stream-json`, and `pice logs --follow --stream-json`.
- Metrics now cover evaluation rows, pass events with cost fields, seam findings, gate decisions, and layer runs.
- npm platform packages include both `pice` and `pice-daemon`; the npm wrapper passes the bundled daemon path to the CLI.

## New And Changed Commands

- `pice init --upgrade` generates `.pice/layers.toml` and contract templates for existing projects.
- `pice layers detect/list/check/graph` manages layer configuration.
- `pice validate --json` validates workflow, layer, and contract config.
- `pice evaluate <plan> --background --wait --timeout-secs <N>` dispatches a background evaluation and waits through a subscribe connection.
- `pice status [feature-id] --follow --stream-json` tails manifest frames.
- `pice logs <feature-id> --follow --stream-json` tails captured provider log frames.
- `pice review-gate --list` and `pice review-gate --gate-id <id> --decision approve|reject|skip` operate pending gates.
- `pice daemon start/status/stop/restart/logs` manages the daemon process.

## Migration Notes

Run:

```bash
pice init
pice init --upgrade
pice layers detect --json
pice layers check --json
pice validate --json
```

Review `.pice/layers.toml` before committing it. Review `.pice/workflow.yaml` if
you opt into custom tiers, budgets, review gates, model overrides, or stricter
parallelism settings.

The daemon stores verification manifests under
`~/.pice/state/{project-hash}/{feature-id}.manifest.json`. Use `PICE_STATE_DIR`
for isolated CI or acceptance runs.

## Breaking Or Operational Changes

- Background commands require daemon mode; `PICE_DAEMON_INLINE=1` intentionally rejects or degrades follow/wait behavior because there is no long-lived process to own subscriptions.
- Provider stdout remains the JSON-RPC channel. Provider logging must go to stderr.
- Evaluation is fail-closed: a layer is not marked passed without provider-backed scoring.
- npm publish is no longer a tag side effect in this branch; it is gated by a manual workflow input.

## Local Metrics And Telemetry

Local SQLite evidence is generated by:

```bash
node scripts/acceptance/metrics-schema-inventory.mjs
```

Expected v0.7.0 local surfaces:

- `gate_decisions`
- `pass_events` with `cost_usd`
- `seam_findings`
- `layer_runs`
- adaptive halt reasons via `halted_by`

`cost_events` is not a separate shipped table in this release; cost is tracked
on `pass_events` and aggregated into evaluation/layer summaries.

Telemetry remains opt-in. Public telemetry must not send code, prompts, file
paths, secrets, or PII.

## Artifact And npm Smoke

The release artifact smoke command is:

```bash
PICE_ARTIFACT_ARCHIVE=/path/to/pice-${target}.tar.gz node scripts/acceptance/release-artifact-smoke.mjs
```

It unpacks the same `.tar.gz` or `.zip` archive shape attached to the GitHub
release, verifies both binaries are present, then runs `pice --version`,
`pice --help`, daemon start/status/stop, and inline `pice validate --json`.

Set `PICE_NPM_PACK_SMOKE=1` in the release workflow after binaries are copied
into npm platform packages to verify local `npm pack` plus install.

## Known Limitations Pending Final Evidence

- The reference projects are intentionally framework-shaped offline fixtures; they do not install external dependencies or contact real databases during CI.
- Quantitative latency and cost-reduction targets are not public release claims unless Task 14 records fresh command output.
- Full native execution for every architecture depends on available GitHub-hosted runners; the CI/release matrices pin macOS x64 to the intended `macos-15-intel` label and macOS arm64 to `macos-15`, alongside Linux x64/arm64 and Windows x64 artifact smoke coverage. Remote GitHub Actions evidence is required before tag approval; this local checkout proves the matrix configuration, not the remote runner result.
- Phase 8 fixture acceptance proves release-flow behavior and seven-layer activation; it does not claim a separate 12-failure-category negative seam-fixture suite or a 100-concurrent-background-evaluation stress run.
